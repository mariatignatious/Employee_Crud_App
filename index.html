Authentication vs Authorization

Authentication vs Authorization

Authentication (AuthN)

Who are you?

  • Verifies the user’s identity
  • Typically involves a password, token, or biometric
  • Happens before authorization
  • HTTP 401 Unauthorized on failure

Authorization (AuthZ)

What can you do?

  • Verifies the user’s permissions
  • Typically involves roles or groups
  • Happens after authentication
  • HTTP 403 Forbidden on failure

Analogy: Authentication is showing your ID at the airport. Authorization is whether your boarding pass lets you into the first-class lounge.

Password Hashing

Step 1: Add Password to Employee Model

We need to store a hashed password for each employee. Add the column to the ORM model:

# models/employee.py

class Employee(Entity):
    __tablename__ = "employee"

    name: Mapped[str] = mapped_column(String(100), nullable=False)
    email: Mapped[str] = mapped_column(String(255), nullable=False, unique=True)
    age: Mapped[int | None] = mapped_column(Integer, nullable=True)
    password_hash: Mapped[str] = mapped_column(String(255), nullable=False)

Never store plain text passwords. We store a hash — the field is called password_hash, not password.

Step 2: The Migration Challenge

Adding a NOT NULL column to a table that already has rows will fail:

ERROR: column "password_hash" of relation "employee" contains null values

Solution: three-step migration

# alembic/versions/c8f05_add_password_and_role_to_employee.py

def upgrade() -> None:
    # 1. Add column as NULLABLE
    op.add_column("employee", sa.Column("password_hash", sa.String(255), nullable=True))

    # 2. Backfill existing rows with a placeholder
    op.execute("UPDATE employee SET password_hash = 'placeholder' WHERE password_hash IS NULL")

    # 3. Make it NOT NULL
    op.alter_column("employee", "password_hash", nullable=False)

Same pattern works for Node.js also: nullable → backfill → not null. Works in any ORM/migration tool.

Why Hash Passwords?

Plain Text Storage

  • Database breach = all passwords exposed
  • Admins can read user passwords
  • Users reuse passwords across sites → cascading compromise

Hashed Storage

  • Database breach = useless hash strings
  • One-way function — cannot reverse
  • Each password has a unique salt
  • Computationally expensive to brute-force

Hashing is one-way: you can verify hash(input) == stored_hash, but you cannot recover the original password from the hash.

Bcrypt

bcrypt is a password hashing algorithm designed to be slow (on purpose).

# Install
uv add bcrypt
# auth/utils.py
import bcrypt

def hash_password(plain: str) -> str:
    return bcrypt.hashpw(plain.encode(), bcrypt.gensalt()).decode()

def verify_password(plain: str, hashed: str) -> bool:
    return bcrypt.checkpw(plain.encode(), hashed.encode())

Salt rounds: gensalt() generates a random salt with a default cost factor of 12. Higher = slower = more secure, but also more CPU per login. The salt is stored as part of the hash string.

Step 3: Hash on Create

Update the employee create flow to accept and hash the password:

employees/schemas.py
class EmployeeCreate(BaseModel):
    name: str
    email: EmailStr
    age: int | None = None
    password: str = Field(min_length=6)
employees/service.py
from auth.utils import hash_password

def create(db, body):
    hashed = hash_password(body.password)
    return employee_repo.create(
        db,
        body.name,
        body.email,
        password_hash=hashed,
        age=body.age,
    )

Important: the plain-text password never reaches the database. The service hashes it before calling the repository.

Login & JWT Tokens

Step 4: Login Endpoint

Create a get_by_email repo method and an auth service:

# employees/repo.py

def get_by_email(db: Session, email: str) -> Employee | None:
    stmt = _employee_stmt().where(
        Employee.email == email,
        Employee.deleted_at.is_(None),
    )
    return db.scalars(stmt).first()
# auth/service.py

def login(db: Session, email: str, password: str) -> str:
    employee = employee_repo.get_by_email(db, email)
    if employee is None:
        raise UnauthorizedException("Invalid email or password")

    if not verify_password(password, employee.password_hash):
        raise UnauthorizedException("Invalid email or password")

    return create_access_token({"id": employee.id, "email": employee.email})

Security note: use the same error message for “email not found” and “wrong password” — don’t reveal which one failed.

What is JWT?

JSON Web Token — a compact, self-contained way to maintain login state between requests.

Structure: Header.Payload.Signature

eyJhbGciOiJIUzI1NiJ9.eyJpZCI6MSwiZW1haWwiOiJhQGIuY29tIn0.dBjftJeZ4CVP-mB92K27uhbUJU1p1r_wW1gFWFOEjXk

PartContainsEncoded?
HeaderAlgorithm (HS256) and token type (JWT)Base64 (readable)
PayloadUser data: id, email, expiry (exp)Base64 (readable)
SignatureHMAC of header + payload using a secret keyEnsures integrity

JWT is NOT encrypted. The payload is Base64-encoded — anyone can decode and read it. Never put sensitive data (passwords, credit cards) in the payload.

Signing: Symmetric vs Asymmetric

Symmetric (HS256)

One shared secret key for signing and verifying.

  • Simple — one key to manage
  • Both issuer and verifier need the same secret
  • Good for single-service apps
jwt.encode(payload, SECRET, algorithm="HS256")
jwt.decode(token, SECRET, algorithms=["HS256"])

Asymmetric (RS256)

Private key signs, public key verifies.

  • Verifiers don’t need the private key
  • Good for microservices / distributed systems
  • More complex key management
jwt.encode(payload, PRIVATE_KEY, algorithm="RS256")
jwt.decode(token, PUBLIC_KEY, algorithms=["RS256"])

We will use HS256 (symmetric) for simplicity. In production with multiple services, consider RS256.

Step 5: JWT Token Generation

# Install
uv add "python-jose[cryptography]"
# auth/utils.py
from datetime import datetime, timedelta, timezone
from jose import JWTError, jwt
from config import settings

def create_access_token(data: dict) -> str:
    to_encode = data.copy()
    expire = datetime.now(timezone.utc) + timedelta(minutes=settings.jwt_expiry_minutes)
    to_encode["exp"] = expire
    return jwt.encode(to_encode, settings.jwt_secret, algorithm=settings.jwt_algorithm)

def decode_access_token(token: str) -> dict | None:
    try:
        return jwt.decode(token, settings.jwt_secret, algorithms=[settings.jwt_algorithm])
    except JWTError:
        return None

Config: JWT_SECRET, JWT_ALGORITHM, and JWT_EXPIRY_MINUTES are loaded from environment variables via Pydantic Settings — never hardcode secrets.

Step 6: Auth Router

# auth/router.py
from fastapi import APIRouter, Depends
from sqlalchemy.orm import Session

from auth import service as auth_service
from auth.schemas import LoginRequest, TokenResponse
from database import get_db

router = APIRouter(prefix="/auth", tags=["Auth"])

@router.post("/login", response_model=TokenResponse)
def login(body: LoginRequest, db: Session = Depends(get_db)):
    token = auth_service.login(db, body.email, body.password)
    return TokenResponse(access_token=token)
# auth/schemas.py
class TokenResponse(BaseModel):
    access_token: str
    token_type: str = "bearer"

Wire it in main.py:

from auth.router import router as auth_router

app.include_router(auth_router)

Sending the Token

The client sends the JWT in the Authorization header on every subsequent request:

POST /auth/login
{ "email": "alice@example.com", "password": "secret123" }

Response:
{ "access_token": "eyJhbGci...", "token_type": "bearer" }

--- All subsequent requests ---

Authorization: Bearer eyJhbGci...

GET /employees

Bearer is the token type as per RFC 6750. It tells the server “the thing after Bearer is the token.”

Step 7: Auth Middleware (FastAPI Dependency)

In FastAPI, we use dependencies instead of Express-style middleware:

# auth/dependencies.py
from fastapi import Depends
from fastapi.security import OAuth2PasswordBearer

from auth.schemas import TokenPayload
from auth.utils import decode_access_token
from exceptions import UnauthorizedException

oauth2_scheme = OAuth2PasswordBearer(tokenUrl="/auth/login")

def get_current_user(token: str = Depends(oauth2_scheme)) -> TokenPayload:
    payload = decode_access_token(token)
    if payload is None:
        raise UnauthorizedException("Invalid or expired token")
    return TokenPayload(**payload)

How it works: OAuth2PasswordBearer extracts the token from the Authorization: Bearer ... header. The get_current_user dependency decodes and validates it.

Step 8: Protecting Routes

Add get_current_user as a dependency to routes that need authentication:

# employees/router.py
from auth.dependencies import get_current_user
from auth.schemas import TokenPayload

@router.get("")
def get_all_employees(
    pagination: dict = Depends(pagination_params),
    db: Session = Depends(get_db),
    _current_user: TokenPayload = Depends(get_current_user),
):
    return employee_service.get_all(db, ...)

Key difference from Express: In Express, middleware is applied to the router and affects all routes. In FastAPI, each route explicitly declares its dependencies — more granular, more visible.

Note: the login endpoint (/auth/login) stays unprotected — users need to be able to log in without a token!

New Exception Classes

We extend our exception hierarchy with auth-specific errors:

# exceptions/__init__.py

class UnauthorizedException(AppException):
    """Authentication required or credentials invalid."""

class ForbiddenException(AppException):
    """Authenticated but not allowed to access this resource."""
# exceptions/handlers.py — add to STATUS_MAP

_STATUS_MAP = {
    NotFoundException:   404,
    ConflictException:   409,
    BadRequestException: 400,
    UnauthorizedException: 401,
    ForbiddenException:    403,
}

Same pattern as before: domain exceptions → global handler → HTTP status codes. No HTTPException in business logic.

The Full Authentication Flow

1. Register → POST /employees with password field → service hashes with bcrypt → stores password_hash

2. Login → POST /auth/login → verify email exists → bcrypt compare → return JWT token

3. Authenticated request → GET /employees with Authorization: Bearer <token>

4. Dependency chain → OAuth2PasswordBearer extracts token → get_current_user decodes JWT → route receives TokenPayload

5. Failure → missing/invalid/expired token → UnauthorizedException → global handler → 401

OAuth2 Password Flow

What is OAuth2?

OAuth2 is a standard authorization framework with multiple “flows” (grant types). We use the Resource Owner Password Credentials flow:

FlowUse Case
Password (what we use)User gives username + password directly to your app
Authorization CodeThird-party login (Google, GitHub) — redirects to provider
Client CredentialsMachine-to-machine (no user involved)
Implicit (deprecated)Was used for SPAs, replaced by Auth Code + PKCE

Password flow is the simplest — the client sends username and password as form data (not JSON), and gets back an access token.

FastAPI’s OAuth2 Utilities

FastAPI provides two classes that implement the password flow out of the box:

OAuth2PasswordRequestForm

Parses login form data: username, password, optional scope.

from fastapi.security import (
    OAuth2PasswordRequestForm,
)

@router.post("/login")
def login(
    form: OAuth2PasswordRequestForm
        = Depends(),
):
    # form.username = email
    # form.password = password

OAuth2PasswordBearer

Extracts Bearer token from Authorization header.

from fastapi.security import (
    OAuth2PasswordBearer,
)

oauth2_scheme = OAuth2PasswordBearer(
    tokenUrl="/auth/login"
)

# Used as a dependency:
token: str = Depends(oauth2_scheme)

Bonus: setting tokenUrl enables the “Authorize” button in Swagger UI — you can test login directly from /docs!

Refactoring Login to OAuth2

Replace our JSON body with the standard OAuth2 form:

Before (JSON body)
@router.post("/login")
def login(
    body: LoginRequest,
    db: Session = Depends(get_db),
):
    token = auth_service.login(
        db, body.email, body.password
    )
After (OAuth2 form)
@router.post("/login")
def login(
    form: OAuth2PasswordRequestForm
        = Depends(),
    db: Session = Depends(get_db),
):
    token = auth_service.login(
        db, form.username, form.password
    )

Why form.username not form.email? The OAuth2 spec defines the field as username. We pass the email as the username — common convention.

Swagger UI Integration

With tokenUrl set, Swagger UI shows a lock icon and an “Authorize” dialog:

How to test from /docs:

  1. Click the Authorize button (top right)
  2. Enter email as username, password as password
  3. Click Authorize
  4. All subsequent “Try it out” requests automatically include the Bearer token

No more copy-pasting tokens! This is one of the biggest benefits of following the OAuth2 spec — tooling support.

Role-Based Access Control

What is RBAC?

Role-Based Access Control — permissions are assigned to roles, users are assigned to roles.

RoleCan ReadCan CreateCan UpdateCan Delete
DeveloperYesNoNoNo
UI / UXYesNoNoNo
HRYesYesYesYes

RBAC simplifies authorization: instead of checking individual permissions, you check the user’s role. New user? Assign a role. Done.

Step 1: Add Role to Employee

Define a Python enum and add it as a column:

# models/employee.py
import enum
from sqlalchemy import Enum

class EmployeeRole(str, enum.Enum):
    UI = "UI"
    UX = "UX"
    DEVELOPER = "Developer"
    HR = "HR"

class Employee(Entity):
    # ... existing fields ...
    role: Mapped[EmployeeRole] = mapped_column(
        Enum(EmployeeRole, name="employeerole"),
        nullable=False,
        server_default=EmployeeRole.DEVELOPER.value,
    )

str, Enum: inheriting from str makes the enum JSON-serializable and comparable with strings. server_default sets the default in the database for existing rows.

Step 2: Migration for Role

# In the same migration file

_ROLE_ENUM = sa.Enum("UI", "UX", "Developer", "HR", name="employeerole")

def upgrade() -> None:
    # ... password_hash steps ...

    # Create the PostgreSQL enum type first
    _ROLE_ENUM.create(op.get_bind(), checkfirst=True)

    # Add column with server_default for existing rows
    op.add_column(
        "employee",
        sa.Column("role", _ROLE_ENUM, nullable=False, server_default="Developer"),
    )

PostgreSQL enum: unlike SQLite, PostgreSQL has native enum types. The migration creates the type first, then uses it in the column. checkfirst=True avoids errors if the type already exists.

Step 3: Include Role in JWT

Add the role to the token payload so we can authorize without hitting the database:

# auth/service.py

def login(db, email, password):
    # ... verify credentials ...

    token = create_access_token({
        "id": employee.id,
        "email": employee.email,
        "role": employee.role.value,
    })
    return token
# auth/schemas.py

class TokenPayload(BaseModel):
    id: int
    email: str
    role: EmployeeRole

Why include role in JWT? Authorization checks happen on every request. Fetching the user from the database each time adds latency. The JWT already contains the role.

Step 4: Role-Based Dependency

Create a dependency factory that checks the current user’s role:

# auth/dependencies.py

def require_role(*roles: EmployeeRole):
    """Return a dependency that checks the user has one of the given roles."""

    def role_checker(
        current_user: TokenPayload = Depends(get_current_user),
    ) -> TokenPayload:
        if current_user.role not in roles:
            raise ForbiddenException(
                "You do not have permission to perform this action"
            )
        return current_user

    return role_checker

Dependency factory pattern: require_role(EmployeeRole.HR) returns a new dependency function with the allowed roles baked in. This is the FastAPI equivalent of Express’s checkRole(role) middleware.

Step 5: Protecting Routes by Role

Use dependencies=[Depends(require_role(...))] on routes that need authorization:

# employees/router.py

@router.post(
    "",
    status_code=status.HTTP_201_CREATED,
    dependencies=[Depends(require_role(EmployeeRole.HR))],
)
def create_employee(body, db):
    ...

@router.put(
    "/{employee_id}",
    dependencies=[Depends(require_role(EmployeeRole.HR))],
)
def update_employee(employee_id, body, db):
    ...

@router.delete(
    "/{employee_id}",
    dependencies=[Depends(require_role(EmployeeRole.HR))],
)
def delete_employee(employee_id, db):
    ...

Read routes (GET) require authentication only (get_current_user). Write routes (POST, PUT, DELETE) require HR role.

Supporting Multiple Roles

Because require_role accepts *roles, you can pass multiple allowed roles:

# Only HR can create employees
dependencies=[Depends(require_role(EmployeeRole.HR))]

# HR and Developers can update
dependencies=[Depends(require_role(EmployeeRole.HR, EmployeeRole.DEVELOPER))]

This is the exercise from the Node.js session — but in Python, *roles (variadic args) gives us this for free. No separate “single role” and “multi-role” versions needed.

The Complete Auth Flow

1. POST /employees (HR creates a user with password + role)

2. POST /auth/login (username=email, password via OAuth2 form → JWT with id, email, role)

3. GET /employees + Bearer token → OAuth2PasswordBearer extracts token → get_current_user decodes → route runs

4. POST /employees + Bearer token → require_role(HR) checks role in JWT → HR: allowed / Developer: 403

Recap & Best Practices

Best Practices

Do

  • Always hash passwords (bcrypt, argon2)
  • Use environment variables for JWT secrets
  • Set token expiry (1h is a good default)
  • Return the same error for “email not found” and “wrong password”
  • Include only necessary data in JWT payload
  • Use Depends() for auth — explicit and testable
  • Separate authentication from authorization

Don’t

  • Store passwords in plain text
  • Put sensitive data in JWT (it’s Base64, not encrypted)
  • Hardcode secrets in source code
  • Use short/weak JWT secrets in production
  • Reveal whether email or password was wrong
  • Skip token expiry (“never expires”)
  • Check roles in business logic — use middleware

Key Takeaways

1. Password Hashing

bcrypt. Hash on create, verify on login. Never store or log plain passwords.

2. JWT Tokens

Stateless authentication. Header.Payload.Signature. Signed (not encrypted). Include user identity + role + expiry.

3. OAuth2 Password Flow

Standard form-based login. FastAPI provides OAuth2PasswordBearer + OAuth2PasswordRequestForm. Enables Swagger UI “Authorize” button.

4. Role-Based Access Control

Roles stored in DB + JWT. require_role(*roles) dependency factory. HR can write, others can only read.

5. FastAPI Pattern

Dependencies > middleware. Domain exceptions > HTTPException. Same clean separation as before.
